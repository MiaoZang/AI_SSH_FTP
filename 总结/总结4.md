## 二、已知问题与限制

### 2.1 当前已知问题

| 问题 | 严重度 | 说明 | 状态 |
|------|--------|------|------|
| SSH fail2ban 封禁 | ⚠️ | 多次密码错误导致 IP 被封 | 等 IP 解封 |
| 异步任务内存存储 | 💡 | 服务重启后任务记录丢失 | 设计如此，非核心需求 |
| FTP 功能较少使用 | 💡 | 大部分操作走 SSH + File API | 保留但不再扩展 |
| WebSocket 使用率低 | 💡 | AI 代理更适合 REST API | 保留 |
| gh release 上传耗时 | 💡 | 大文件 (~63MB) 上传慢 | 可后台运行 |

### 2.2 安全注意事项

| 风险 | 缓解措施 |
|------|----------|
| 密码明文存储在 config.yaml | 部署在内网，限制访问 |
| API 无认证 | 仅绑定内网 IP，或用防火墙限制 |
| Base64 ≠ 加密 | 仅防注入，非安全措施 |
| SSH 权限过大（root） | 可用受限用户 + sudoers |

### 2.3 架构限制

| 限制 | 原因 | 未来方案 |
|------|------|----------|
| 单目标服务器 | 设计为代理单台 SSH 服务器 | 多目标支持需重构 |
| 无 API 认证 | 部署在可信网络 | 可加 API Key 中间件 |
| 无 TLS/HTTPS | 内网使用 | 可用 Nginx 反向代理 |
| 异步任务无持久化 | 简单设计 | 可加 SQLite/BoltDB |

## 三、关键设计决策记录

### 3.1 为什么用 HTTP API 而非直接 SSH？

**决策：** AI Agent 通过 HTTP API 间接操作服务器，而非直接 SSH

**原因：**
1. Windows PowerShell 没有 `sshpass`，无法非交互传递密码
2. AI Agent 处理 SSH 交互（密码提示、菜单选择）不可靠
3. HTTP 请求有标准化的错误处理、超时控制
4. Base64 编码避免了 shell 转义问题
5. 所有操作有日志记录（可审计）

### 3.2 为什么 GET API 用于 SSH 执行？

**决策：** `GET /api/ssh/exec?cmd=BASE64` 作为 PowerShell 推荐调用方式

**原因：**
1. PowerShell 的 `Invoke-RestMethod` 构造 JSON body 时，`$` 和 `"` 被自动展开
2. `curl.exe` 在 PS 中的 JSON 转义极其麻烦（需要三层转义）
3. GET 请求通过 URL 参数传递，完全避开 JSON 构造
4. Base64 编码确保 URL 安全

**权衡：** GET 请求有 URL 长度限制（~2KB），超长命令仍需 POST

### 3.3 为什么配置文件改为可选？

**决策：** v1.7.0 让 config.yaml 不存在时也能启动

**原因：**
1. AI 部署时创建配置文件是额外步骤，容易出错
2. 环境变量方式更适合 CI/CD 和 Docker
3. 仍然支持配置文件，向后兼容

### 3.4 为什么异步任务存在内存中？

**决策：** `sync.Map` 存储异步任务，不用数据库

**原因：**
1. 异步任务是临时的，查看结果后即可丢弃
2. 服务重启后旧任务已无意义
3. 避免额外数据库依赖，保持二进制单文件部署

### 3.5 为什么 FTP 和 SSH File 功能并存？

**决策：** 保留 FTP API，同时新增 HTTP File API

**原因：**
1. FTP API 是 v1.0 遗留，某些场景仍需要
2. HTTP File API 通过 SSH/SFTP 实现，更灵活
3. HTTP Upload 支持自动解压，FTP 不支持
4. 两者共存不冲突，逐步弃用 FTP

## 四、经验教训

### 4.1 PowerShell 兼容性

```
教训：永远不要假设 PowerShell 和 Bash 的行为一致

具体问题：
- $变量 会被 PS 展开，即使在双引号字符串中
- @"..."@ heredoc 对换行、空格极其敏感
- Invoke-RestMethod 和 Invoke-WebRequest 行为不同
- curl 在 PS 中是 Invoke-WebRequest 的 alias，不是 curl.exe
- Connection: keep-alive 会导致 PS 挂起
```

### 4.2 AI Agent 与交互式脚本

```
教训：AI Agent 完全无法操作交互式菜单

具体问题：
- AI 通过 Web Terminal 操作时，无法可靠识别菜单选项
- 交互式 read 提示会导致 AI 陷入死循环
- 密码提示的超时行为不可控

最佳实践：
- 为 AI 提供纯 CLI 模式（命令行参数）
- 所有交互都可通过参数跳过
- 文档中明确标注「AI 用」vs「人用」
```

### 4.3 SSH 连接管理

```
教训：SSH Session 不是线程安全的

具体问题：
- 多个请求共用一个 Session 会导致数据混乱
- 解决方案：每次 Exec() 创建新的 Session
- 连接池管理比想象中复杂

最佳实践：
- 每个请求创建独立的 SSH Session
- 设置合理的连接超时
- 处理 SSH 断连重连
```

### 4.4 GitHub Release 大文件

```
教训：gh release create 上传大文件需要很长时间

具体问题：
- 两个二进制 ~63MB，上传需要 2+ 分钟
- 命令没有进度条，看起来像卡住了
- 但实际上是成功的

最佳实践：
- 后台运行 gh release 命令
- 或先创建空 release，再分别 upload 文件
```

### 4.5 fail2ban 与自动化

```
教训：多次 SSH 密码错误会触发 fail2ban Ban

具体问题：
- 自动化脚本密码输入失败后重试，累积触发 ban
- Ban 后所有 SSH 连接都被拒绝

最佳实践：
- 使用 SSH Key 认证代替密码
- 或在 fail2ban 中白名单开发 IP
```
