# ai-ssh-ftp-proxy 技能改进建议（第二期）

> 更新时间：2026-03-09 16:42
> 基于 xpay-3.1 实际部署全流程中遇到的新问题总结

---

## 一、本次使用中发现的新问题

### 1. PowerShell 多行命令与 Base64 编码的冲突
- **问题**：在 PowerShell 中使用 `@" ... "@` 构造多行 bash 命令再 Base64 编码时，PowerShell 的 heredoc 解析经常出错（`UnexpectedToken` 报错），导致命令根本没有发到服务器。
- **根因**：PowerShell 的 `@"..."@` 对特殊字符（`$`、`"`）非常敏感，而 bash 命令中大量使用这些字符。
- **影响**：需要将复合命令拆成多条单行命令逐条发送，效率低下。
- **建议**：SKILL.md 中应提供 **PowerShell 专用的调用模板**，明确使用 `@'...'@`（单引号 heredoc）避免变量展开问题。

### 2. 长时间命令的超时与输出丢失
- **问题**：Maven 编译（`mvn clean package`）需要 50+ 秒，`apt install` 需要更久。HTTP API 调用如果不设置足够的 `-TimeoutSec`，会在中途超时断开，但服务器上的命令仍在继续执行。
- **影响**：无法确认命令是否真正执行成功，需要额外发送检查命令验证。
- **建议**：
  - API 应支持 **异步执行模式**：返回 `task_id`，客户端可通过 `GET /api/ssh/task/{id}` 轮询状态和输出。
  - 或支持 **流式响应**（chunked transfer），实时返回 stdout。

### 3. 输出在 PowerShell 中乱序/混合
- **问题**：多条 PowerShell 命令快速执行时，`$stdout` 变量可能残留上一次的输出内容，导致当前命令的 `Write-Host $stdout` 显示的是上一条的结果。
- **根因**：PowerShell 的变量作用域在同一终端会话中共享，且 `Invoke-RestMethod` 的返回值被覆盖时机不确定。
- **影响**：调试时非常混乱，需要反复执行验证命令才能确认真实结果。
- **建议**：SKILL.md 应提供 **独立封装的 PowerShell 函数**，每次调用自带变量隔离：

```powershell
function Invoke-SSHCommand {
    param([string]$Command, [string]$Server = "http://154.222.29.99:48891")
    $b64 = [Convert]::ToBase64String([Text.Encoding]::UTF8.GetBytes($Command))
    $body = @{command = $b64} | ConvertTo-Json
    $resp = Invoke-RestMethod -Uri "$Server/api/ssh/exec" -Method POST -ContentType "application/json" -Body $body -TimeoutSec 300
    $out = [Text.Encoding]::UTF8.GetString([Convert]::FromBase64String($resp.stdout))
    $err = if ($resp.stderr) { [Text.Encoding]::UTF8.GetString([Convert]::FromBase64String($resp.stderr)) } else { "" }
    return @{ stdout = $out; stderr = $err; exit_code = $resp.exit_code }
}
# 使用：
$result = Invoke-SSHCommand "ls -la /"
Write-Host $result.stdout
```

### 4. 文件上传 API 的 extract 参数缺少文档
- **问题**：上传 tar.gz 并自动解压到目标目录时，用了 `-F "extract=true"` 参数，但 SKILL.md 中虽然提到了该功能，格式说明不够清晰（`path` 既要 Base64 编码，又要区分文件路径和目录路径）。
- **建议**：SKILL.md 应增加完整的 **上传并解压示例**：

```bash
# 上传 tar.gz 并自动解压到 /www/wwwroot/app/
DEST=$(echo -n "/www/wwwroot/app/" | base64)
curl -X POST http://SERVER:48891/api/file/upload \
  -F "file=@local.tar.gz" \
  -F "path=$DEST" \
  -F "extract=true"
```

### 5. 无法批量执行命令/脚本
- **问题**：部署过程需要执行 10+ 条关联命令（创建目录、修改文件、编译、启动等），每条都要单独 HTTP 请求 + Base64 编码 + 解码输出，非常繁琐。
- **建议**：
  - 新增 `/api/ssh/script` 端点，支持上传并执行整个 bash 脚本。
  - 或支持在 `/api/ssh/exec` 中传入多条命令数组，按顺序执行并分别返回结果。

## 二、与已有建议的对比

| 第一期（部署阶段） | 第二期（使用阶段） |
|---|---|
| 交互式菜单无法操作 | PowerShell 调用模板缺失 |
| 文档缺少免交互说明 | 文档缺少上传解压完整示例 |
| 端口冲突无提示 | 长命令超时无异步机制 |
| Windows 无 sshpass | 输出乱序需函数封装 |
| 环境变量配置 / health 端点 | 批量命令/脚本执行支持 |

第一期聚焦「怎么把服务装上」，第二期聚焦「装上后怎么高效使用」。

## 三、优先级排序

| 优先级 | 改进项 | 原因 |
|---|---|---|
| 🔴 **P0** | PowerShell 封装函数 | 当前 AI 每次调用都要写 6 行样板代码 |
| 🔴 **P0** | 异步执行模式 | 编译等长任务无法可靠获取结果 |
| 🟡 **P1** | 上传解压示例补全 | 文件部署是核心场景 |
| 🟡 **P1** | 批量命令/脚本执行 | 减少 HTTP 往返次数 |
| 🟢 **P2** | 输出乱序问题说明 | 属于使用注意事项 |
